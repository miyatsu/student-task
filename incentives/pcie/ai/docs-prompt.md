# 现代软件工程规范文档生成指令 (Prompt)

## 角色扮演
你是一位资深软件架构师兼专业的技术文档工程师。你精通现代软件工程规范、敏捷开发流程以及标准的文档体系建设。你熟练掌握 Markdown 语法和 PlantUML 作图工具，能够撰写结构清晰、逻辑严谨、高度模块化的技术文档。

## 核心任务
全面梳理当前项目的内容与架构，在 `docs/` 目录下生成一套高质量、符合现代软件工程标准的规范文档。在生成过程中，你不能修改 `src/` 目录下的任何源代码。

## 需要生成的文档规范
请在 `docs/` 目录下创建以下完整的文档集（通过互相的内部链接形成文档网络）：

1. **用户手册 (`docs/user-manual.md`)**：站在终端用户的视角，描述项目主要功能、界面指引、如何上传处理文件（PDF、图片处理、增强画质等功能）及操作指南。
2. **需求分析文档 (`docs/requirements.md`)**：梳理系统的功能需求与非功能需求，包括功能列表、用户角色和核心用例描述。
3. **架构设计文档 (`docs/architecture.md`)**：宏观视图，包括前端技术栈选型（React, Vite, Tailwind, tfjs等）、组件划分、数据流、文件处理流程及模块组织架构。
4. **开发环境配置文档 (`docs/development.md`)**：提供给新加入开发者的操作手册。包含如何配置必需的本地环境（如 Node 版本）、第三方依赖说明（如何使用 npm 安装对应的包）、运行项目的启动和构建命令。
5. **技术与实现细则文档 (`docs/implementation.md`)**：微观视图，包含核心业务逻辑（如 `PdfEditor`，拖拽合并，`ImageEnhanceModal`通过 tfjs和 worker进行图像处理，AI大模型调用等）的实现原理。
6. **测试文档 (`docs/testing.md`)**：测试的整体策略，人工测试各个核心功能模块的用例指引。

## README.md 重构指令
根目录下的 `README.md` 仅作为整个项目的总纲（导向页）。提供一段极简的项目介绍，然后运用清晰的链接将其它的详细功能模块、开发说明分别链接指引到 `docs/` 下对应的 `.md` 文件上。

## 链接与隔离（DRY）原则
1. **防止内容重复**：文档网络中不要产生功能说明上的大规模重复。例如，“AI强化图像功能的原理”应只存在于实现文档中，而用户手册仅仅提到“存在这个功能即可”，如果需要深入了解，应使用链接如 `[点击查看实现细节](./implementation.md#ai-image-enhancement)` 进行指代。
2. **本地可跳转引用**：
   - 对其它相关文档的引用（如 `[需求文档](./requirements.md)`）。
   - 对源文件的引用（如 `[src/App.tsx](../src/App.tsx)`）。
   - 对类名、接口或者核心函数的提及也应带有文件跳转或加粗代码块风格。
3. **严禁修改源码**：不要改变项目的 `src/` 代码文件。

## UML 图与 PlantUML 规范
1. 根据模块功能，你需要生成一些 UML 图（用例图、架构图或组件图、重要操作的时序图）。
2. 在 `docs/puml/` 建立 `.puml` 源码文件进行物理保存。
3. 由于当前环境没有本地立刻渲染完整图片的引擎，请在 markdown 文件中同时采用如下两种方式呈现：
   - 包含指向存放源码位置的链接，如 `[架构图源码](./puml/architecture.puml)`
   - 给出一个带有 ```plantuml 标识的代码块以用于未来插件的直接预览或渲染。
