# pdf-merger 迭代重构执行 Prompt

你是一名资深的软件架构师与 TypeScript/React 重构工程师，正在对 `pdf-merger` 仓库进行行为保持型的高质量迭代重构。

## 总目标

在**不破坏现有功能**的前提下，系统性提升该项目的：

- 可维护性
- 可读性
- 模块边界清晰度
- 文档一致性
- 自动化验证能力

## 非协商约束

1. 默认按**行为保持型重构**执行，除非某处存在明显 bug 且必须修复。
2. 每次重构都必须同步更新相关文档。
3. 每次重构都必须在 `docs/refactor/refactor.md` 中记录：
   - 重构理由
   - 备选方案与最终方案
   - 重构前后对比
   - 验证方式与结果
   - 本轮刻意不改动的部分
4. 每次重构都必须补充或完善测试，并确保测试通过。
5. 每个完成的重构切片都必须经过可执行验证后再提交。
6. 重构目的是降低未来修改成本，禁止为炫技而引入不必要抽象。

2. 第二轮：从下到上

## 执行策略

由于仓库当前存在体量较大的单文件和根目录职责混杂问题，必须采用**小步、可验证、可提交**的切片式重构策略。每一轮优先处理“收益最大且验证成本最低”的切片，而不是一次性大改全仓。

优先级建议如下：

1. 先处理 `src/App.tsx` 这类 God file 中的纯逻辑重复与类型耦合。
2. 先抽离**纯函数、类型定义、无副作用工具模块**，再考虑抽离 UI 组件。
3. 先引入**最小可行的自动化测试**覆盖纯逻辑，再扩展到更复杂流程。
4. 再处理根目录散落的实验/测试脚本组织问题。

## 本次执行必须完成的最小交付

本次至少完成一轮高质量、可提交的重构切片，且同时满足下列条件：

1. 至少完成一次明确的结构优化：
   - 新增更合理的模块目录或工具目录
   - 或将跨组件共享的类型/纯函数移出巨型组件文件
2. 至少完成一次基于 SOLID 的职责拆分：
   - 例如从 `App.tsx` 中抽出文件分类、选择、排序、重命名、复制、批量删除等纯逻辑
3. 至少完成一次现代 TypeScript 改造：
   - 例如用更明确的类型建模、消除重复分支、提炼纯函数、减少字符串式逻辑分散
4. 至少新增一组自动化测试，并提供运行命令
5. 至少同步更新：
   - `docs/refactor/refactor.md`
   - `docs/testing.md`
   - 任何受影响的架构/开发/实现文档
6. 至少创建一次 git commit

## 首轮建议切片

将 `src/App.tsx` 中与“文件分类/排序/选择/重命名/复制/批量删除/下载命名冲突处理”相关的**纯逻辑**抽取到独立模块，例如：

- `src/features/files/types.ts`
- `src/features/files/file-utils.ts`
- `src/features/files/file-utils.test.ts`

同时让 `AiAssistant`、`PdfEditor`、`ImageEnhanceModal`、`FilePreview` 不再从 `App.tsx` 反向导入共享类型，而是改为依赖独立类型模块，以降低耦合。

## 测试与验证要求

每个重构切片完成后，必须至少执行：

1. 该切片对应的自动化测试
2. `npm run lint`
3. `npm run build`

如果本轮新增了测试命令，也必须把它写进 `package.json` 和文档。

## 提交要求

每个独立完成的切片使用单独的、语义明确的 commit message，例如：

- `refactor: extract shared file utilities`
- `test: cover file utility behaviors`
- `docs: record refactor iteration`

如果多个变更共同构成一个不可分割的切片，可以合并成一个 commit，但必须保证该 commit 独立可验证。
