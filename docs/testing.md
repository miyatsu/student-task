# 测试文档

为了保证这套轻量级但功能复杂的多功能文件处理逻辑保持高度可用性，应当在版本迭代之前经历以下测试。当前策略已经同时覆盖纯逻辑回归、关键应用层回归以及主干流程的人工验证。

## 1. 测试策略
鉴于系统的纯前端运行特点：
1. **组件与应用层自动化测试**：当前已经为 `src/features/files/file-utils.ts` 建立纯逻辑回归测试，并为文件列表组件、`PdfEditor` 页操作流程、`gemini.ts` 运行时配置加载建立了应用层回归测试。
2. **浏览器 E2E 原理验证**：保障 Canvas 操作不产生跨域死锁或者内存逸出（OOM）。
3. **集成验证**：重点在跨模块之间流转的文件，例如是否在不同区域依然保持正确的名字以及拓展后缀名。这需要人工 Checklist 校验。
4. **实验 / 诊断脚本**：`scripts/experiments/` 下的脚本用于环境探测、库能力验证和手工联调，不属于自动化回归测试树，也不会被 `npm test` 执行。

### 1.1 当前自动化测试入口

```bash
npm test
```

该命令当前会顺序运行以下两层自动化测试：

```bash
npm run test:logic
npm run test:app
```

- `npm run test:logic` 会运行 `src/features/files/file-utils.test.ts`，用于验证文件分类、选择切换、排序、重命名、复制、批量删除以及 Zip 导出重名处理逻辑。
- `npm run test:app` 会运行当前的 Vitest 应用层回归测试，包括：
	- `ImageFilesSection`、`PdfFilesSection`、`WordFilesSection` 的列表交互、行内上移/下移、排序头 UI、批量操作事件路由，以及图片 / Word 转 PDF 进度展示。
	- `PdfEditor` 的页删除、页抽取与“禁止删除全部页面”行为。
	- `gemini.ts` 的运行时配置加载、缓存与客户端创建逻辑。
	- `server.ts` 中压缩命令生成、旧版 `.doc` 文本提取、压缩等级映射与运行时配置读取所依赖的服务端纯函数。

兼容性说明：`npm run test:ui` 目前仍保留为 `npm run test:app` 的别名，方便沿用旧命令。

### 1.2 实验脚本与自动化测试的边界

`scripts/experiments/` 当前按用途拆分为：

- `diagnostics/`：环境、CLI、第三方库 API 与能力探测。
- `manual/`：需要本地服务或手工观察结果的联调脚本。
- `spikes/`：一次性技术验证与可行性试验。

这些脚本虽然名字里带有 `test-`，但语义上更接近“开发期验证工具”，不应与真正的自动化回归测试混放到统一的 `test/` 树中。

## 2. 人工测试大纲及执行 Check List

开发者或测试人员按以下标准开展预发布检验。

### 模块一：全局文件组织 (DND与排序)
| 用例编号 | 用例名称     | 操作动作                                                                        | 预期结果                                                                        | 状态 |
|----------|-------------|--------------------------------------------------------------------------------|---------------------------------------------------------------------------------|------|
| TC-01    | 类型混合空降 | 同时按住一本 PDF，一张 `jpg`，一个 `docx` 后，拖放至 App 工作虚线框中。          | 文件必须顺利分发到对应的 3 个独立的（Image/PDF/Word）区块内部并提供对应缩略状态。 | [ ]  |
| TC-02    | 跨文件删除   | 在各个分类中分别选中若干文件，点击红色的清空/删除选定。                         | 仅仅只有那些 Checkbox=true 的文件在试图内消失，其余保留不变。                      | [ ]  |
| TC-03    | 名称自然排序 | 在同一分组内上传 `file1.pdf`、`file2.pdf`、`file10.pdf`，点击“NAME”排序。         | 名称排序应识别数字块，顺序显示为 `file1`、`file2`、`file10`，而不是字典序。          | [ ]  |
| TC-04    | 行内顺序微调 | 在任意分组中对中间文件连续点击“上移 / 下移”按钮。                                  | 文件应按点击方向与相邻项交换位置，且当前分组的排序状态被清空，改为手动顺序。         | [ ]  |
| TC-05    | 排序控件可发现性 | 首次进入页面，不做任何操作，观察 `NAME / DATE / SIZE` 所在区域。                 | 排序区域应呈现为带图标的胶囊按钮组，用户无需 hover 或试错也能感知“这里可以排序”。   | [ ]  |
| TC-06    | 全局排序切换 | 点击文字“SIZE (大小)”标签的排序按钮。                                            | 组内按从达到小升序排列对应的大小展示；点击第二次，改为升序或者降序，方向图标更改。 | [ ]  |

### 模块二：AI 图像超级强化处理 (`ImageEnhanceModal`)
| 用例编号 | 用例名称     | 操作动作                                                                        | 预期结果                                                                        | 状态 |
|----------|-------------|--------------------------------------------------------------------------------|---------------------------------------------------------------------------------|------|
| TC-07    | 特大尺寸载入 | 上传并点击 4K 以上分辨率或 5MB 以上图片的魔棒特效。                              | 由于 Patch-Size 的保护，等待虽然漫长（超 30s），但浏览器不得崩溃或者显示白屏抛出异常。 | [ ]  |
| TC-08    | 滑块边界检测 | 在完成增强的对比状态下，用鼠标将游标滑至最左方边界并停止，再滑动至右侧边缘并松开。 | 左方停止时应完全展示原图（含水印），右方全幅显示增强质感。不应该因为游标脱模影响布局。  | [ ]  |
| TC-09    | 中断生命周期 | AI 模型处于 Loading 状态（旋转缓冲），立刻点击底界面的 "Cancel" 中止窗口组件。     | 等待数秒不应阻碍程序的正常销毁；模型推理的上下文被回收且在原页面内无任何不兼容假死。 | [ ]  |
